**LIFT LOG**

Journal Module — Initial Specification

*Version 1.0 — March 2026*

*Status: Confirmed design — ready for implementation*

# 1. Purpose of This Document

This document is the initial specification for the Lift Log Journal
module. It covers concept, UX flow, prompt behaviour, the user profile
system, data model guidance, and access controls. It is intended as a
briefing for implementation and as source material for core-ref-v2 (not
yet written).

All decisions in this document are confirmed unless explicitly noted
otherwise. The data model section (Section 5) is guidance only — the
implementing developer should review it against the current architecture
and confirm the most appropriate approach before building.

# 2. Navigation — Coach Button Clarification

The existing app has a persistent 'COACH' button at the bottom centre
of the modules screen. This button is visible in the screenshot shared
during the design session. It is a floating label/button in the bottom
navigation area, currently opening the pre-session wellness check-in
conversation.

This button is being retained in its current position but redesigned as
an entry point to the Coach screen hub mentioned in section 10 of this
document. The Coach screen contains: the daily wellness check-in (greyed
out once complete), the user profile, and cross-exercise progress
(backlog). The navigation placement — bottom centre persistent button
— does not change.

> *ℹ The Coach screen hub is a separate workstream from the Journal
> module. This document focuses on the Journal. The Coach screen
> redesign should be specified separately when that work begins.*

# 3. Journal Module — Concept

## 3.1 What it is

A daily reflective space with three AI-generated prompts. The journal
knows who the user is, what they are working toward, and what has been
on their mind recently. It becomes more useful over time without
becoming heavier — it maintains a compact, evolving user profile
rather than reading a full archive of past entries each day.

The module is contextually aware. One prompt is always grounded in
training progress and how the user feels about it. The other two draw
from the user profile, recent wellness scores, and past journal
responses. All three prompts are generated by Claude Sonnet.

## 3.2 The three daily prompts

Three prompts are generated in a single Sonnet call when the user first
opens the journal each day. They are not regenerated on subsequent opens
— the same three prompts are shown until the user submits or the
calendar day ends.

| | |
|---|---|
| **Prompt 1 — Progress & Reflection** | Grounded in training data and how the user feels about their progress. Focuses on the emotional and motivational dimension of training — goals, confidence, effort, direction. Only pulls in specific workout data once the user has tracked a minimum of 5 sessions on a given exercise. Before that threshold, this prompt asks more open questions about training goals and what the user is hoping to achieve and how they feel personally about progress. It is not a mechanical workout summary — it should prompt genuine reflection. |
| **Prompt 2 — Profile-informed** | Drawn from the user profile, recent wellness scores, and recent journal responses. Reflects on what has been on the user's mind, emerging themes, or areas of life that seem relevant. For the first 5 total journal entries (while the profile builds), this prompt asks 'get to know you' style questions rather than drawing on a profile that does not yet have depth. Wellness scores inform this prompt — patterns in sleep, stress, or feed scores over time should surface here when relevant. |
| **Prompt 3 — Open / Broader** | A broader or more open question. May relate to life priorities, goals beyond training, or themes surfaced by the onboarding conversation or recent entries. More varied in character than Prompts 1 and 2. |

> *ℹ Prompt logic for early users: the 5-session threshold for workout
> data and the 5-entry threshold for profile depth are independent
> checks. Both should be evaluated at prompt generation time and the
> appropriate version of each prompt selected accordingly.*

**Prompt refresh**

Each prompt has a refresh icon next to it. Tapping refresh regenerates
that single prompt — the other two are unchanged. This allows the user
to swap out a prompt that does not resonate without discarding the whole
set. A refreshed prompt is stored in place of the original for that day.

## 3.3 What the prompt generator reads

The Sonnet call that generates the three prompts receives the following
context:

- Today's wellness scores — sleep, feed, stress from daily_wellness
  (null-safe — absent scores are handled gracefully)

- Recent workout data — last 3–5 sessions across exercises where
  session count for each exercise that data is drawn from is >= 5.
  Exercises below the threshold are excluded from prompt 1 data but
  their existence is noted.

- The current user profile — full profile text, described in Section 4

- Recent journal entries — the last 3 days (i.e. 9 entries total)
  (prompt used + response text)

- Journal entry count — total entries to date, used to determine
  whether early-user prompt variants apply

> *ℹ The prompt generator never reads the full journal entry history.
> All historical context is compressed into the user profile. This keeps
> the context window bounded regardless of how long the user has been
> journalling.*

## 3.4 Entry format and UX flow

**The journal screen**

The journal screen opens with three prompts, each with its own text box
beneath it. A submit button sits at the bottom of the screen. A back
chevron ( \< ) in the top left returns the user to the modules screen
without submitting.

| | |
|---|---|
| **Submit button — no entries** | Greyed out and non-functional. The user must type something in at least one text box before submit activates. |
| **Submit button — partial entries** | Active. If the user taps Submit without having filled in all three prompts, a confirmation dialog appears: 'You haven't responded to all prompts. You won't be able to come back to answer them today. Submit anyway?' Two options: Submit / Go back. |
| **Submit button — all entries** | Active. Submits immediately with no dialog. |
| **In-progress save** | If the user has started typing but has not submitted (i.e. they have returned to the module screen through a '\<' in the top left of the screen or have closed the app), their draft responses are saved to localStorage keyed to that day. On reopening the module, their draft is restored. The draft is cleared at the start of a new calendar day when fresh prompts are generated. |
| **Already submitted today** | If the user has already submitted their journal for the day, the module opens to a confirmation screen: 'You've already recorded your journal for today. Come back tomorrow for fresh prompts.' No editing of submitted entries. |
| **No free-text outside prompts** | There is no general free-text field. The user can only respond to the three prompts. If they do not wish to respond to any prompt, they can leave and return another day. The module does not force engagement. |
| **Maximum response length** | 100 words per prompt. Enforced in the UI with a word count indicator. Enforced at the Worker on submission. |

**Daily reset**

The module resets each calendar day. Fresh prompts are generated on
first open. In-progress localStorage drafts from the previous day are
discarded. The submitted entry from the previous day is preserved in D1
and visible in the entry history view.

## 3.5 Journal entry history

A history view is part of the v1 module. Accessible from within the
journal screen (e.g. a History button or link below the prompts area).
Displays past entries in reverse chronological order: date, the prompt
used, and the user's response for each. Read-only — past entries
cannot be edited.

> *ℹ This is a v1 feature, not backlog.*

# 4. The User Profile

## 4.1 What it is

A living, AI-maintained summary of the user. Generated and updated by
Claude Sonnet. Not a list of facts — a curated prose document that
captures who the user is, what they are working toward, and what
patterns have been emerging across their training, wellness, and
reflections.

The profile is the connective tissue of the personalisation layer. It is
what allows prompt generation to feel contextually aware rather than
generic. Over time it also informs the pre-session wellness coach
framing, making the whole app feel like it knows the user.

## 4.2 Profile structure

Stored as freeform prose with consistent section headings. Sonnet always
populates each section. The user reads it as a flowing document — not
a form with fields.

| | |
|---|---|
| **Training focus** | Current training priorities, goals, block focus, anything notable about where the user is in their programme. |
| **Work and life context** | What the user does professionally, current demands, anything relevant to energy and stress patterns. |
| **Life priorities** | What matters beyond training — relationships, health goals, broader wellbeing, stated values. |
| **Wellness patterns** | Observations derived from wellness score history — recurring patterns in sleep, feed, or stress scores. For example: sleep scores consistently lower mid-week, stress elevated during certain periods. This section is populated once enough wellness data exists to identify patterns (approximately 2 weeks of check-ins, 2 weeks in time elapsed not necessarily 14 total entries in case the user misses a day or a few, can mention that the user is inconsistent with their check ins if so). Before that threshold it is left sparse. |
| **Recent themes** | What has come up repeatedly in journal entries over recent weeks. Emotional or situational patterns worth carrying forward. |
| **Notable patterns** | Cross-domain observations — connections between training performance, wellness scores, and what the user has written. Populated progressively as data accumulates. |

## 4.3 Profile update flow

The profile is updated every time the user submits a journal entry. The
Worker makes a background Sonnet call after entry submission — the
user does not wait for it. The call receives:

- The current profile text

- The new journal entry (all responses submitted)

- Today's wellness scores

- Recent wellness score history — last 14 days, to support the
  Wellness patterns section

Sonnet returns an updated profile. The update is evolutionary — it
refines and adds, it does not rewrite from scratch. Sections shift as
patterns emerge and fade. The updated profile replaces the current one
in D1.

> *ℹ Profile update is a discrete, lightweight background call. It does
> not block the user experience. If it fails, the previous profile is
> retained and the update retried on the next submission.*

## 4.4 Profile storage location

The user profile is stored as a column on the existing users table in
D1, not as a separate table. A profile_text TEXT column is added to
users. This keeps identity and profile co-located and avoids an
additional join for every prompt generation call.

> *ℹ This is a suggestion. The implementing developer should confirm
> this is appropriate given the current schema and any size constraints
> on D1 TEXT columns. If the profile is expected to grow large, a
> separate table may be more appropriate.*

## 4.5 Profile visibility and editing

The user will be able to access their profile from the Coach screen
(once this has been built but in v1 this functionality does not exist
yet, only the ability to respond to journal prompts in the journal
module). It is displayed as readable prose with section headings
visible.

An Edit button opens a text editor with the full profile content. The
user will be able to correct, add, or remove anything. Submitting the
edit writes the edited text directly back to D1 — no AI call is made
on manual edits. The next profile update (triggered by a journal entry
submission) will use the edited version as its starting point.

> *ℹ This preserves the auditability principle. The AI's interpretation
> of the user is transparent and correctable.*

## 4.6 Profile seeding — the onboarding conversation

The user profile starts empty. On the first time the user opens the
journal tab, before any prompts are shown, a one-time onboarding
conversation runs. This is implemented in v1 — it is not a backlog
item.

The onboarding conversation is a distinct AI interaction with a warmer,
more exploratory tone than the daily wellness coach. It is not a scoring
exercise. It asks about:

- The user's work and professional life

- Their training goals — what they are working toward and why

- What else matters to their wellbeing beyond training

- Anything else they want the app to understand about them

The conversation runs for approximately 8–12 exchanges. It is powered
by Claude Sonnet with a bespoke prompt (separate from the coaching
prompts). After it completes, it never runs again for that user.

The onboarding responses are used to generate the initial user profile
via a Sonnet call. On the day the onboarding conversation is completed,
no journal prompts are shown. Instead, the user sees a dedicated screen:

> *"Thanks for introducing yourself. I've made a note of what matters
> to you. Come back tomorrow for your first journal prompts."*

The exact wording should be reviewed before implementation — it should
feel warm and human, not transactional.

# 5. Data Model — Implementation Guidance

The following is design guidance for the data model. It is not a final
schema instruction. The implementing developer (referred to as 'Code'
in the project) must review this against the current D1 schema, the
existing Worker architecture, and future optionality before confirming
the approach. The goal is the simplest model that meets the requirements
without closing off future doors.

## 5.1 Suggested new tables

**journal_entries**

One row per submitted journal entry. Stores all three prompt/response
pairs for a given day. The UNIQUE constraint on (userId, date) enforces
the one-submission-per-day rule at the database level — not just in
application logic.

| | |
|---|---|
| **id** | INTEGER — auto-incrementing primary key |
| **userId** | TEXT — references users.id |
| **date** | TEXT — YYYY-MM-DD — UNIQUE with userId |
| **prompt_1_text** | TEXT — the prompt shown to the user |
| **response_1** | TEXT — the user's response, or NULL if not answered |
| **prompt_2_text** | TEXT |
| **response_2** | TEXT — nullable |
| **prompt_3_text** | TEXT |
| **response_3** | TEXT — nullable |
| **createdAt** | TEXT — ISO timestamp of submission |

> *ℹ Storing the prompt text alongside the response means the entry is
> self-contained — the history view does not need to re-generate or
> look up what the prompt was. This is the rationale for this structure
> vs a normalised prompt/response pair model.*

**journal_prompts**

One row per day per user. Stores the three generated prompts once
created. On subsequent opens of the journal on the same day, the Worker
returns the stored row rather than calling Sonnet again.

| | |
|---|---|
| **id** | INTEGER — auto-incrementing primary key |
| **userId** | TEXT — references users.id |
| **date** | TEXT — YYYY-MM-DD |
| **prompt_1** | TEXT — the training/progress prompt |
| **prompt_2** | TEXT — the profile-informed prompt |
| **prompt_3** | TEXT — the open/broader prompt |
| **refresh_count_1** | INTEGER — number of times prompt 1 was refreshed. Default 0. |
| **refresh_count_2** | INTEGER — number of times prompt 2 was refreshed. Default 0. |
| **refresh_count_3** | INTEGER — number of times prompt 3 was refreshed. Default 0. |
| **generatedAt** | TEXT — ISO timestamp |

> *ℹ If a user refreshes a single prompt, the relevant column is updated
> in place and the corresponding refresh_count is incremented. The history
> of what prompt was originally generated is not retained — the
> journal_entries row captures what was actually shown at submission time.
> Refresh counts are a product signal: high counts on a given prompt type
> indicate users are finding those prompts unrelatable.*

**journal_onboarding_conversations**

One row per message exchange in the onboarding conversation. Follows
the same pattern as `coaching_conversations` — stored message by
message in D1 to support tab-close recovery. Separate table from
`coaching_conversations` to allow independent querying and retention.

| | |
|---|---|
| **id** | INTEGER — auto-incrementing primary key |
| **userId** | TEXT — references users.id |
| **messages** | TEXT — full conversation history as JSON array |
| **status** | TEXT — 'in_progress' / 'complete' |
| **createdAt** | TEXT — ISO timestamp |
| **updatedAt** | TEXT — ISO timestamp |

> *ℹ On tab close mid-onboarding, the user returns to find the
> conversation exactly where they left it. The Worker checks
> journal_onboarding_conversations status on journal module open to
> determine whether onboarding is still in progress.*

**Onboarding completion screen**

On the day the onboarding conversation completes, no journal prompts
are shown. The user sees a dedicated screen with a warm, forward-looking
message. Confirmed wording (to be reviewed before ship):

> *"All set. Your first prompts will be waiting tomorrow."*

The journal tile on the modules screen is accessible that day but opens
to this screen rather than prompts. From the next calendar day, prompts
are generated normally.

---

## 5.2 Suggested change to existing tables

**users table — add profile_text column**

Rather than creating a separate user_profile table, the profile is added
as a TEXT column on the existing users table. This avoids an additional
join on every prompt generation call and keeps identity and profile
co-located.

| | |
|---|---|
| **profile_text** | TEXT — the full profile as freeform prose. NULL until the onboarding conversation completes. |
| **profile_updated_at** | TEXT — ISO timestamp of last profile update. Helps the Worker determine whether a profile exists yet. |

> *ℹ If the users table is expected to be queried frequently without
> needing profile text (e.g. auth checks), and the profile grows large,
> a separate table may be more efficient. Code should assess this
> against actual usage patterns.*

## 5.3 Suggested new Worker endpoints

| | |
|---|---|
| **GET /journal/status?date=x** | Returns whether today's journal has been submitted, and whether the onboarding conversation has been completed. Used on journal module open to determine which screen to show. |
| **GET /journal/prompts?date=x** | Returns today's prompts. If not yet generated, calls Sonnet, stores them, and returns. If already generated, returns from D1. If the onboarding conversation has not been completed, returns a flag indicating onboarding is required instead. |
| **POST /journal/prompts/refresh** | Regenerate a single prompt. Body: { date, prompt_number (1/2/3) }. Calls Sonnet for one new prompt, updates the relevant column in journal_prompts, returns the new prompt text. |
| **POST /journal/entry** | Save a submitted journal entry. Body: { date, prompt_1_text, response_1, prompt_2_text, response_2, prompt_3_text, response_3 }. Before saving, the Worker runs keyword detection across all response fields. If flagged content is detected, the entry is not saved, the profile update is not triggered, and the fixed redirect message is returned to the client. See Section 11.3. If no flags, the entry is saved and async profile update triggered. |
| **GET /journal/history** | Return all past journal entries for the user, newest first. Each entry includes date, all three prompt/response pairs. |
| **GET /profile** | Return the user's current profile_text from the users table. |
| **PUT /profile** | Update profile with manually edited text. Body: { profile_text }. Writes directly — no AI call. |
| **POST /journal/onboarding/message** | Send a user message in the onboarding conversation and receive the next coach message. Follows the same pattern as the coaching module's message endpoint. When the conversation completes, the Worker generates the initial profile and stores it. |
| **GET /journal/onboarding/status** | Returns whether the onboarding conversation has been completed for this user. |

# 6. AI Model

All AI calls in the journal module use Claude Sonnet. This applies to:

- Daily prompt generation

- Single prompt refresh

- Profile update (triggered by entry submission)

- Onboarding conversation

- Initial profile generation (from onboarding responses)

The coaching module (pre- and post-session wellness scoring) continues
to use Claude Haiku. The journal module requires more nuance and
qualitative sensitivity than the scoring task — Sonnet is the right
choice here.

> *ℹ Model is specified in worker/src/index.js. The journal module will
> use a separate callSonnet() function or a model parameter on the
> existing callClaude() function. Code should determine the cleanest
> implementation.*

The Journal module is in beta. The Journal tile on the modules screen is

When the module is opened to all users, the frontend render condition is

> *ℹ The beta access list is hardcoded in only the frontend for now.
> When the module is opened to all users, the beta reference is
> removed.*

# 8. Onboarding Conversation — Prompt Guidance

The onboarding prompt is a bespoke system prompt, separate from the
CORE_PROMPT and phase prompts used by the wellness coach. It should be
written and stored in the same location as the coaching prompts
(worker/src/index.js) and documented in coaching-prompts-v1.md or a new
journal-prompts-v1.md file.

Key requirements for the onboarding prompt:

- Identify the AI as the Lift Log journal assistant at the start

- Warm and curious tone — this is the only conversation that should
  feel like getting to know someone

- No scoring. No data collection beyond building context.

- Ask about work, training goals, life priorities, what else matters
  to wellbeing

- Open questions only — do not ask the user to rate, score, or
  categorise anything

- 8–12 exchanges maximum. Do not drag it out.

- End gracefully — confirm that the profile has been set up and that
  the user should return tomorrow

The exact prompt wording is the responsibility of the product owner
(Toby), who may make edits once beta implementation is complete. This
will require real world use to see how the onboarding conversation feels
and a manual review of the prompt behind it.

> *ℹ Unlike the wellness coach prompts, the onboarding conversation does
> not use tool_use for structured output. It ends when the AI determines
> the conversation is complete, and the Worker then makes a separate
> Sonnet call to generate the initial profile from the full conversation
> history.*

# 11. Guardrails — Safety and Scope

## 11.1 Why this section exists

The journal module is a more open interaction than the wellness coaching
check-in. The coaching conversation is tightly scoped — it asks
specific questions, runs for a fixed number of exchanges, and exists
solely to produce scores. The journal is different. It invites
reflection on training, work, stress, goals, and life more broadly. That
openness is its value. It is also where the risk is.

People using a reflective journalling tool may share things that go
beyond what a fitness app is equipped to respond to. This is not a
theoretical concern — it is a known failure mode that has been
designed against from the start of this project, and the motivation for
that caution comes from direct personal experience of AI dependency
during difficult periods.

The guardrails in this section are non-negotiable. They are not features
that can be switched off or relaxed as the module matures. They are the
conditions under which the module is safe to ship at all.

## 11.2 What the journal is not

The journal module is a reflective writing tool. It is not a mental
health tool, a therapeutic service, a crisis resource, or a substitute
for professional support. This distinction must be clear in the product
itself — not just in documentation.

The onboarding conversation and daily prompts are designed to help the
user reflect on their training, their life, and their wellbeing in
service of building a useful personalisation layer. They are not
designed to explore trauma, provide emotional support, or respond to
distress. When those things surface — and over time, in a journalling
context, they may — the module must recognise them and respond
correctly.

## 11.3 Keyword protocol — immediate stop

The journal module inherits and extends the keyword protocol from the
wellness coaching module. If the user writes anything suggesting
self-harm, crisis, hopelessness, suicidal ideation, or acute mental
health distress, the following applies:

**In the onboarding conversation:** The AI stops immediately. It does
not continue the conversation, generate a profile, or respond in a way
that extends the interaction. It outputs a fixed, non-AI-generated
redirect message and ends the session. The onboarding is marked
incomplete — it can be restarted when the user is ready.

**In daily journal prompt responses:** Keyword detection runs at the
Worker level on submission, before the entry is saved and before any
profile update is triggered. If flagged content is detected, the entry
is not saved, the profile update is not triggered, and the user is shown
the fixed redirect message.

The fixed redirect message for the journal module is:

*"Something in what you've written suggests you might benefit from
talking to someone directly. Lift Log isn't the right tool for this —
but these services are free and available now:*

*Samaritans — call 116 123 (24/7)*
*Shout — text SHOUT to 85258 (24/7)*
*NHS 111 — call 111, select option 2 (24/7)*
*CALM — call 0800 58 58 58 (5pm–midnight)*

*Your previous entries are safe — come back when you're ready."*

This message is fixed in the Worker. It is never AI-generated. It does
not vary. The reason for this, as established in the coaching module
design, is that AI-generated responses in potential crisis situations
are unpredictable and carry clinical risk. A fixed response does exactly
one thing consistently: it signposts the user to appropriate help and
ends the interaction.

**Support resources screen**

A support resources screen is accessible at all times via a button in
the top right of the modules screen. It is not hidden behind the journal
module — it is visible to all users from day one, regardless of beta
access status. The button should be unobtrusive but clearly findable; a
small icon or text link is sufficient.

The screen displays the following resources:

| **Resource** | **Contact** | **Availability** |
|---|---|---|
| Samaritans | Call 116 123 or visit samaritans.org | Free, 24/7 |
| Shout | Text SHOUT to 85258 | Free, 24/7 |
| NHS 111 | Call 111, select option 2 | Free, 24/7 |
| CALM | Call 0800 58 58 58 or webchat at thecalmzone.net | Free, 5pm–midnight, 365 days a year |
| Mind | mind.org.uk/information-support/guides-to-support-and-services/crisis-services | Directory of local services |

The screen includes a short header: *"If you're struggling, you don't
have to deal with it alone. These services are free, confidential, and
available any time."*

The resources listed are UK-based. When the platform opens to
international users at Stage 2, this screen will need to be revisited to
either localise resources by region or link to an international
directory such as findahelpline.com.

> *ℹ The fixed redirect message in the Worker references this screen
> ("there is a list of resources in the app") — both must be
> implemented together. Neither is useful without the other.*

## 11.4 Scope constraint on the onboarding conversation

The onboarding conversation is the most open-ended AI interaction in the
app. It asks about work, goals, life priorities, and wellbeing. It is
designed to be warm and exploratory. That tone must not drift into a
support or counselling register.

The onboarding system prompt must explicitly constrain the AI from:
offering advice, providing emotional support, engaging with disclosed
personal difficulties beyond acknowledging them and redirecting, or
extending the conversation in response to distress signals. If the user
discloses something difficult, the AI acknowledges it briefly and moves
on — it does not explore it. The keyword protocol applies here with
the same force as everywhere else in the module.

## 11.5 No therapeutic framing

The journal prompts, profile sections, and any AI-generated content in
the module must not use therapeutic language or framing. This means: no
prompts that invite the user to process grief, trauma, or acute
emotional difficulty. No profile sections that characterise the user's
mental state in clinical terms. No language that implies the app is
tracking or monitoring the user's psychological wellbeing.

The wellness patterns section of the user profile (Section 4.2) observes
patterns in scored data — sleep, stress, feed — not in disclosed
emotional content. It does not summarise how the user has been feeling
emotionally. That distinction is enforced in the profile update prompt.

## 11.6 AI disclosure

The journal module identifies itself as AI at every interaction point
where a user could mistake it for a human. This includes:

- The first message of the onboarding conversation

- Any screen that presents AI-generated prompts (a static label is
  sufficient — "Today's prompts are generated by AI")

- The profile view (a note that the profile is AI-maintained)

This is consistent with the disclosure principle applied across the
coaching module and is non-negotiable.

## 11.7 The dependency risk

A journalling tool that gets to know the user over time, generates
personalised prompts, and maintains a profile of who they are carries a
specific dependency risk that a simple data logger does not. The more
useful it becomes, the more the user may come to rely on it as a place
to process their inner life.

This is not inherently a problem. But it becomes one if the tool
positions itself as a substitute for human connection, professional
support, or therapeutic relationship — even implicitly, through tone
and framing.

The design choices that mitigate this risk are already embedded in this
specification: the fixed keyword redirect, the scope constraint on the
onboarding, the prohibition on therapeutic framing, the clear AI
disclosure. The prompt wording — which is the product owner's
